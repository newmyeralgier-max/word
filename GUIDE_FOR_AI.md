# GUIDE_FOR_AI.md

Инструкция для AI-ассистента (Devin/любой другой), работающего с этим репозиторием.

---

## 1. О чём этот репозиторий

Это рабочее пространство для двух Word-документов:

- **`WORD/data/Исходник/БР.docx`** + **`БР_титульник.docx`** — выпускная работа.
- **`WORD/data/Готово/Уч пособ Матлаб ПРАВИЛЬНО3.docx`** — учебное пособие по Matlab/Simulink.

Цель — приводить эти документы под ГОСТ 7.32-2017 по мере появления новых задач от пользователя.

**Других задач в репо нет.** Если кажется, что нужно добавить какой-то новый функционал, не относящийся к этим двум файлам, — это, скорее всего, мусор.

## 3. Как ты должен работать

### 3.1. Стартовый чек-лист

1. Прочитай `WORD/data/задание.md` — там текущая задача от пользователя.
2. Определи тип задачи (см. §3.5 — три воркфлоу).
3. Прочитай `PROBLEM_CATALOG.md` — возможно, симптом уже описан.
4. Если описан — запусти соответствующий скрипт через `run_all.py --only NN`.
5. Если не описан — добавь новый скрипт `pipeline/NN_*.py` + запись в каталог + PR.

### 3.2. Принципы

- **Один скрипт = одна проблема.** Не клади в `04_wrap_figures.py` логику нумерации формул.
- **Идемпотентность.** Запуск скрипта дважды не должен ничего ломать.
- **Интерфейс одинаковый.** Все фиксеры принимают `--input X.docx --output Y.docx`. Без исключений.
- **Без зависимостей между скриптами.** Каждый сам по себе. Оркестрирует только `run_all.py`.
- **Бэкапы автоматически.** `run_all.py` кладёт бэкап в `WORD/data/backups/<имя>_pre_run.docx`.

### 3.3. Что НЕ делать

- ❌ Не создавай разовые `analyze_*.py` / `check_*.py` / `verify_*.py` в корне репо или в `pipeline/`. Если нужна диагностика — пиши inline через `python -c "..."` или ad-hoc, **НЕ коммить**.
- ❌ Не создавай новые папки в корне (кроме `WORD/data/` структуры пользователя). Скрипты — в `pipeline/` или подпапке `pipeline/<тема>/`.
- ❌ Не редактируй `.docx` файлы вручную (cat/echo/awk). Только через скрипты.
- ❌ Не создавай «универсальный мега-скрипт» вместо отдельных шагов — пользователь явно отказался.
- ❌ Не используй `git add .` — добавляй конкретные файлы.
- ❌ Не пуш в main напрямую. Всегда через PR с веткой `devin/<timestamp>-<тема>`.
- ❌ Не делай коммитов с .tmp файлами, бэкапами, внутренними логами.

### 3.4. Что точно НЕ ТРОГАТЬ

- `WORD/data/Исходник/БР.docx`
- `WORD/data/Исходник/БР_титульник.docx`
- `WORD/data/Готово/Уч пособ Матлаб ПРАВИЛЬНО3.docx`
- `WORD/data/Готово/БР_GOST.docx`
- `WORD/data/Скриншоты/`
- `WORD/data/задание.md` (если пользователь сам не попросил отредактировать)

Скрипты `pipeline/*` не трогают входной файл — они читают `--input` и пишут `--output`. Менять файлы пользователя в исходном виде нельзя; результат всегда отдельным файлом или через бэкап.

### 3.5. Три типа задач (воркфлоу)

#### A. «Приведи готовый DOCX к ГОСТ»

Пример: «Поправь ПРАВИЛЬНО3 — там пропали номера страниц.»

```bash
python pipeline/run_all.py \
    --input  "WORD/data/Готово/Уч пособ Матлаб ПРАВИЛЬНО3.docx" \
    --output "WORD/data/Готово/Уч пособ Матлаб ПРАВИЛЬНО3.docx" \
    --only 09,10,11
```

#### B. «Я дал MD-черновик — собери из него .docx»

Пример: пользователь даёт `WORD/data/Исходник/draft.md` и просит «сделай мне нормальный документ».

```bash
# 1. Собрать docx из MD
python pipeline/md_to_docx/build_docx.py \
    -i WORD/data/Исходник/draft.md \
    -o WORD/data/Готово/draft.docx \
    --fast

# 2. Прогнать через ГОСТ-фиксеры
python pipeline/run_all.py \
    -i WORD/data/Готово/draft.docx \
    -o WORD/data/Готово/draft.docx
```

#### C. «У меня есть готовый DOCX — допиши в него этот MD-кусок»

Пример: пользователь даёт `new_chapter.md` и просит «добавь это в БР, не пересоздавая файл».

```bash
# 1. Дописать MD в конец существующего docx
python pipeline/md_to_docx/build_docx.py \
    -i WORD/data/Исходник/new_chapter.md \
    -o WORD/data/Готово/БР_GOST.docx \
    --append WORD/data/Готово/БР_GOST.docx \
    --fast

# 2. Привести к ГОСТ (особенно номера формул и разрывы страниц)
python pipeline/run_all.py \
    -i WORD/data/Готово/БР_GOST.docx \
    -o WORD/data/Готово/БР_GOST.docx
```

> **Важно для воркфлоу C**: `--append` НЕ пересоздаёт файл, а добавляет в конец. Титульник, оглавление, существующие главы — не трогаются. После append всегда прогоняй `run_all.py --only 07` (перенумерация формул) чтобы новые формулы получили правильные номера в общей сквозной нумерации.

## 4. Типичный воркфлоу для новой задачи

```bash
# 1) Создать рабочую ветку
git checkout main && git pull
git checkout -b devin/$(date +%s)-task-name

# 2) Прочитать задание
cat WORD/data/задание.md

# 3) Применить нужный воркфлоу (A/B/C из §3.5)

# 4) Срендерить в PDF и посмотреть глазами (если нужна визуальная проверка)
soffice --headless --convert-to pdf \
    "WORD/data/Готово/Уч пособ Матлаб ПРАВИЛЬНО3.docx" \
    --outdir /tmp/render
pdftoppm -r 100 /tmp/render/*.pdf /tmp/render/page

# 5) Если нашёл новую проблему — добавь скрипт `pipeline/NN_<...>.py` и запись в каталог
# 6) Создать PR
```
