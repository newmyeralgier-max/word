# GUIDE_FOR_AI.md

Инструкция для AI-ассистента (Devin/любой другой), работающего с этим репозиторием.

---

## 1. О чём этот репозиторий

Это рабочее пространство для двух Word-документов:

- **`БР.docx`** + **`БР_титульник.docx`** — выпускная работа.
- **`data/Уч пособ Матлаб ПРАВИЛЬНО3.docx`** — учебное пособие по Matlab/Simulink.

Цель — приводить эти документы под ГОСТ 7.32-2017 по мере появления новых задач от пользователя.

**Других задач в репо нет.** Если кажется, что нужно добавить какой-то новый функционал, не относящийся к этим двум файлам, — это, скорее всего, мусор.

## 3. Как ты должен работать

### 3.1. Стартовый чек-лист

1. Прочитай `data/задание.md` — там текущая задача.
2. Прочитай `PROBLEM_CATALOG.md` — возможно, твоя «новая» проблема уже описана.
3. Если описана — запусти соответствующий скрипт из `pipeline/` через `run_all.py --only NN`.
4. Если не описана:
   - Воспроизведи симптом (LibreOffice → PDF → визуально).
   - Найди причину в OOXML (распакуй `.docx` как zip, ищи в `word/document.xml`/`settings.xml`/`footer*.xml`).
   - Напиши **новый** скрипт `pipeline/NN_<имя>.py` (НЕ в корне, НЕ в новой папке).
   - Добавь его в `STEPS` в `pipeline/run_all.py`.
   - Опиши проблему в `PROBLEM_CATALOG.md` и в таблице `pipeline/README.md`.
   - Создай PR.

### 3.2. Принципы

- **Один скрипт = одна проблема.** Не клади в `04_wrap_figures.py` логику нумерации формул.
- **Идемпотентность.** Запуск скрипта дважды не должен ничего ломать.
- **Интерфейс одинаковый.** Все скрипты принимают `--input X.docx --output Y.docx`. Без исключений.
- **Без зависимостей между скриптами.** Каждый сам по себе. Оркестрирует только `run_all.py`.
- **Бэкапы автоматически.** `run_all.py` сам кладёт бэкап в `data/backups/<имя>_pre_run.docx`.

### 3.3. Что НЕ делать

- ❌ Не создавай разовые `analyze_*.py` / `check_*.py` / `verify_*.py` в корне репо. Если нужна диагностика — пиши inline через `python -c "..."` или ad-hoc, **НЕ коммить**.
- ❌ Не создавай новые папки. Все скрипты — только в `pipeline/`. Все доки — в корне.
- ❌ Не редактируй `.docx` файлы вручную (cat/echo/awk). Только через скрипты.
- ❌ Не создавай «универсальный мега-скрипт» вместо отдельных шагов — пользователь явно отказался от этого.
- ❌ Не используй `git add .` — добавляй конкретные файлы.
- ❌ Не пуш в main напрямую. Всегда через PR с веткой `devin/<timestamp>-<тема>`.
- ❌ Не делай коммитов с .tmp файлами, бэкапами, внутренними логами.

### 3.4. Что точно НЕ ТРОГАТЬ

- `БР.docx`
- `БР_титульник.docx`
- `data/Уч пособ Матлаб ПРАВИЛЬНО3.docx`
- `data/БР_GOST.docx`
- `Practice/` (matlab-источники)
- `data/Скриншоты/`
- `data/задание.md` (если пользователь сам не попросил отредактировать)

Скрипты `pipeline/*` не трогают входной файл — они читают `--input` и пишут `--output`. Менять файлы пользователя в исходном виде нельзя; результат всегда отдельным файлом или через бэкап.

## 4. Типичный воркфлоу для новой задачи

```bash
# 1) Создать рабочую ветку
git checkout main && git pull
git checkout -b devin/$(date +%s)-task-name

# 2) Прогнать существующий пайплайн на текущем файле
python pipeline/run_all.py \
    --input  "data/Уч пособ Матлаб ПРАВИЛЬНО3.docx" \
    --output "data/Уч пособ Матлаб ПРАВИЛЬНО3_GOST.docx"

# 3) Срендерить в PDF и посмотреть глазами
soffice --headless --convert-to pdf "data/Уч пособ Матлаб ПРАВИЛЬНО3_GOST.docx" --outdir /tmp/render
pdftoppm -r 100 /tmp/render/Уч пособ Матлаб ПРАВИЛЬНО3_GOST.pdf /tmp/render/page

# 4) Если нашёл новую проблему — добавь скрипт `pipeline/13_<...>.py` и запись в каталог
# 5) Прогнать ещё раз через run_all
# 6) Создать PR
```

## 5. Окружение

Должно быть установлено:

```bash
pip install python-docx lxml
apt-get install -y libreoffice poppler-utils  # для рендера в PDF и PNG
```

Python ≥3.10. Тестировалось на Linux, но скрипты кросс-платформенные (нет хардкода путей).
