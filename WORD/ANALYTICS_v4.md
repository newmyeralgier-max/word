## 3. АРХИТЕКТУРНЫЕ ПРОБЛЕМЫ (Phase 4 step 5: Question Architecture)

### 3.1. Форматер делает 3 разные вещи

1. **Детекция** (_classify) — определяет тип параграфа
2. **Трансформация** (_handle_*) — меняет контент (объединяет, удаляет, перезаписывает)
3. **Форматирование** (_set_run_font, _clear_indents_and_set) — меняет визуальные свойства

Проблема: шаг 2 не нужен. ГОСТ-форматер должен ТОЛЬКО форматировать, не трансформировать.

### 3.2. Нет разделения "безопасные" vs "опасные" операции

Все фиксы применяются безоткатно. Нет dry-run, нет верификации что контент сохранён.

### 3.3. extract_title.py — костыль

Вместо того чтобы НЕ ломать титульник, форматер ломает его, а потом восстанавливает. Это двойная работа с двойным риском.

### 3.4. 32 функции с пересекающейся логикой

_handle_heading, _handle_structural_h1, _handle_figure_caption, _handle_table_caption — все делают одно и то же: меняют стиль, выравнивание, шрифт. Но каждая добавляет свою "магию".

## 4. НОВЫЙ ПОДХОД: КОНСЕРВАТИВНЫЙ ФОРМАТЕР

### Принцип: "PRIMUM NON NOCERE" (прежде всего — не навреди)

**Что форматер ДЕЛАЕТ:**
1. Меняет стили параграфов (Normal → Heading, etc.)
2. Меняет шрифты/размеры runs (TNR 14pt)
3. Меняет выравнивание параграфов (JUSTIFY, CENTER)
4. Меняет отступы параграфов (first_line_indent, space_before/after)
5. Меняет tab stops (формулы)
6. Ставит ГОСТ-границы таблицам и jc=center
7. Ставит ГОСТ-поля секций
8. Восстанавливает секции и титульник из оригинала

**Что форматер НЕ ДЕЛАЕТ:**
1. ❌ Не удаляет параграфы (никогда)
2. ❌ Не перезаписывает текст runs (никогда, кроме minor: точка в конце заголовка)
3. ❌ Не объединяет заголовки
4. ❌ Не удаляет пустые абзацы
5. ❌ Не удаляет OMML-параграфы
6. ❌ Не перемещает элементы в XML
7. ❌ Не создаёт новые параграфы (кроме TOC)

### Зоны документа

| Зона | Что делаем | Не делаем |
|------|-----------|-----------|
| Титульник (до ВВЕДЕНИЕ) | Восстановить из оригинала | Ничего не меняем |
| Основная зона | Стили, шрифты, отступы, выравнивание | Не трогаем контент |
| После ЗАКЛЮЧЕНИЯ | Восстановить из оригинала | Ничего не меняем |

### Зачем "восстановить из оригинала" вместо "не трогать"?

Потому что setup_gost_styles() меняет стили глобально — Normal, Heading 1-3. Эти стили наследуются параграфами титульника. Если мы не восстановим титульник, стили съедут.

**Альтернатива**: setup_gost_styles() НЕ трогать базовый стиль Normal, а создавать НОВЫЕ стили (GOST_Normal, GOST_Heading 1, etc.) и назначать их параграфам основной зоны. Тогда титульник не пострадает.

## 5. ПЛАН ПЕРЕРАБОТКИ (v4.0)

### Фаза 1: Создать валидационный скрипт (30 мин)

**Цель**: Автоматическая проверка что форматер не убил контент.

**Файл**: `/mnt/d/1. Project/Word/WORD/execution/validate_formatter.py`

```python
"""validate_formatter.py — Проверка что форматер не потерял контент.
Запуск: python validate_formatter.py original.docx formatted.docx
Возвращает 0 если всё ок, 1 если есть потери."""
```

Проверки:
1. Все параграфы из оригинала присутствуют в GOST (по тексту)
2. Кол-во таблиц совпадает (±0)
3. Кол-во изображений совпадает (±0)
4. Кол-во OMML-формул совпадает (±0)
5. Нет параграфов где p.text.strip() != "" но p.text пустой в GOST
6. Отчёт: сколько/какие параграфы потеряны, какие появились

### Фаза 2: Отключить опасные операции (30 мин)

**Что отключить (закомментировать):**
1. ФИКС I — удаление серий пустых абзацев (L1067-1083)
2. ФИКС 14 — удаление хвостовых параграфов (L1085-1116)
3. ФИКС 16 — объединение заголовков (L1118-1141)
4. ФИКС 32 — удаление пустых Heading (L1143-1160)
5. _handle_table_caption — удаление следующего параграфа (L865)
6. _format_formula — перезапись runs для OMML (L349-366)

**Что оставить:**
- Все _handle_* функции — но без удаления параграфов
- _set_run_font — но без перезаписи run.text
- _normalize_tables — jc=center, границы
- Восстановление секций и титульника

### Фаза 3: Переработать setup_gost_styles (1 час)

**Проблема**: setup_gost_styles() модифицирует стандартные стили (Normal, Heading 1-3), которые наследуются параграфами титульника.

**Решение**: Создать КАСТОМНЫЕ стили (GOST_Body, GOST_Heading1, GOST_Heading2, GOST_Heading3), которые:
- Основаны на Normal (base_style)
- Имеют все ГОСТ-параметры (шрифт, размер, интервал)
- Назначаются ТОЛЬКО параграфам основной зоны
- НЕ наследуются параграфами титульника (т.к. титульник остаётся в Normal)

**Файл**: `word_utils.py` → `setup_gost_styles()`

### Фаза 4: Переработать _format_formula (30 мин)

**Для OMML-формул** (m:oMath):
- НЕ трогать runs вообще
- Только: p.alignment = CENTER, tab stops (RIGHT tab для номера)
- Проблема: номер формулы в runs, тело — в m:oMath. Tab stop справа выровняет номер.

**Для текстовых формул** (без m:oMath, с "=" и номером):
- НЕ перезаписывать runs
- Только: p.alignment = CENTER, tab stops
- Если есть номер (N.N) — правая табуляция

### Фаза 5: Переработать _handle_table_caption (30 мин)

**Текущая проблема**: Находит следующий параграф с текстом, сливает в капшон, УДАЛЯЕТ.

**Новое поведение**:
1. Капшон "Таблица 2.2" → стиль Normal, CENTER, TNR, bold для "Таблица N.N"
2. Следующий параграф с описанием → НЕ трогать (оставить как есть)
3. НЕ удалять параграфы
4. Ренейминг "Табл." → "Таблица" — только если пользователь не указал --legacy

### Фаза 6: Переработать _handle_figure_caption (15 мин)

Аналогично таблицам:
1. "Рис. 4.2" → стиль Normal, CENTER, TNR
2. НЕ удалять параграфы
3. Ренейминг "Рис." → "Рисунок" — только если не --legacy

### Фаза 7: Переработать _handle_empty (15 мин)

**Текущее**: Удаляет page break в пустых параграфах → переносит на предыдущий.

**Новое**: НЕ удалять. Пустые параграфы — оставить как есть. Если между капшоном и таблицей 3 пустых абзаца — пусть будут. Word сожмёт их при рендеринге.

### Фаза 8: Тестирование и валидация (30 мин)

1. Запустить форматер на БР.docx
2. Запустить validate_formatter.py
3. Проверить что 0 потерь контента
4. Визуальная проверка в Word

## 10. ВЫВОД

Текущий форматер — жертва "fix cascade": каждый фикс порождал новый баг, каждый баг порождал новый фикс. 15 .remove() вызовов и 11 перезаписей run.text — это не форматер, это контент-терминатор.

**Решение**: Принцип "PRIMUM NON NOCERE". Форматер форматирует — он не удаляет, не перезаписывает, не объединяет. Если что-то визуально не идеально (2 пустых строки вместо 1) — это терпимо. Потеря контента — не терпимо.
